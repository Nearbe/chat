# Инструкции для работы с проектом Chat

## Основные принципы
1. **Язык документации**: Все комментарии в коде, KDoc и описания коммитов должны быть на **русском языке**.
2. **Стиль кода**: 
   - Используйте Swift-стандарты (CamelCase для типов и свойств).
   - Обязательно добавляйте Docstrings (///) для всех публичных методов, структур и классов.
   - Разделяйте код на логические блоки с помощью `// MARK: -`.
3. **Архитектура**: 
   - Проект использует MVVM + SwiftData.
   - Сетевой слой реализован через `HTTPClient` и `NetworkService`.
   - Для работы со стримингом используется `SSEParser`.

## Работа с API LM Studio
- Используйте v1 API LM Studio.
- Модели находятся в `Models/LMStudio/`.
- Базовый URL и таймауты хранятся в `AppConfig`.

## Авторизация
- Авторизация привязана к устройству через `DeviceConfiguration` и `DeviceIdentity`.
- Токены хранятся только в Keychain через `KeychainHelper`.
- Не сохраняйте секреты в `UserDefaults`.

## Проверка качества и работоспособности (Workflows)
Для обеспечения качества кода и автоматизации процесса используйте два основных скрипта, разделенных по ролевой модели.

### 1. Технический флоу (check.sh)
Предназначен для ИИ-помощника или разработчика. Позволяет быстро получить информацию о тестах и состоянии проекта. **Настроен на работу с симулятором без code signing, чтобы избежать запросов пароля.**
```bash
chmod +x check.sh
./check.sh "Ваше описание изменений"
```
Что делает скрипт:
- Генерирует проект (XcodeGen).
- Проверяет стиль кода (SwiftLint).
- Генерирует ресурсы (SwiftGen).
- Собирает проект в Debug конфигурации.
- Запускает все Unit и UI тесты с выводом сводки.
- **Коммитит все изменения и делает push в удаленный репозиторий.**

### 2. Продуктовый флоу (ship.sh)
Используется для доставки готового продукта главному потребителю (боссу, тестеру). Должен запускаться **ПОСЛЕ** успешного выполнения `check.sh`.
```bash
chmod +x ship.sh
./ship.sh
```
Что делает скрипт:
- Собирает проект в Release конфигурации.
- Устанавливает и запускает приложение на устройстве **Saint Celestine**.
*(Опирается на результат работы check.sh)*

## Поддержание актуальности (Maintenance)
**Критическое правило для ИИ**: При любых изменениях в структуре проекта, ключевых компонентах или архитектуре, вы ОБЯЗАНЫ проверить и при необходимости обновить следующие файлы:

## Ограничения разработки (Personal Team License)
Проект разрабатывается с учетом ограничений бесплатной/персональной команды Apple Developer (Personal Team). При проектировании новых функций необходимо учитывать НЕДОСТУПНОСТЬ следующих инструментов и сервисов:
1. **CloudKit**: Использование облачного хранилища данных Apple невозможно.
2. **Push Notifications**: Удаленные уведомления не поддерживаются.
3. **In-App Purchases**: Тестирование и реализация покупок внутри приложения ограничены.
4. **App Groups**: Обмен данными между приложениями и расширениями через группы ограничен.
5. **Apple Pay**: Интеграция платежной системы невозможна.
6. **Siri & Game Center**: Эти сервисы недоступны для персональных команд.
7. **Срок действия профиля**: Профили подготовки (provisioning profiles) для бесплатных аккаунтов истекают через 7 дней.
8. **Лимит устройств**: Ограничение на количество зарегистрированных устройств для тестирования (обычно до 3).
9. **Распространение**: Невозможно распространение через TestFlight или App Store.

**Важное примечание по автоматизации**: Чтобы избежать постоянных запросов пароля Keychain при работе `check.sh`, скрипт настроен на использование симулятора с флагами `CODE_SIGNING_ALLOWED=NO`. Физические устройства (используемые в `ship.sh`) требуют обязательного подписания, что всегда влечет за собой запрос пароля в macOS.
